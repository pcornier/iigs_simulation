# Core Timing Improvement Plan — Beam-Racing Accuracy (textfunk and friends)

**Goal:** get the FPGA-friendly RTL close enough to real-hardware CPU↔video timing that
beam-racing demos (textfunk.po first; more to follow) render without tearing, while keeping
all regression tests and ROM self-tests passing.

**Status of inputs (all verified this session):**
- The **65C816 core is cycle-accurate and functionally correct** — SingleStepTests harness
  (`SingleStepTests/sim_main.cpp`, committed) shows 504/512 opcode files perfect; the 8
  outliers are known *test-fixture* bugs. So the tearing is **NOT** in the CPU microcode.
- The tearing is in the **wall-clock timing model** — specifically the relationship between
  `rtl/clock_divider.v` (CPU PH2 generation) and `rtl/video_timing.v` (the video beam).

---

## 1. Root cause (confirmed)

We run **two independent timebases** that are never re-locked to each other:

| | Video side | CPU side |
|---|---|---|
| Module | `rtl/video_timing.v` | `rtl/clock_divider.v` |
| Counter | `hcount` 0..911, `hidx`/`hsub` (Mega II char) | `ph0_counter` 0..13, `scanline_ph0_ctr` 0..64 |
| Clock | `clk_vid` (28M) + `ce_pix` | `clk_14M` |
| Line length | 912 ticks (stretch absorbed at `hcount==HWL`, `video_timing.v:96`) | 65·14 + 2 = 912 (stretch added at `scanline_ph0_ctr==64`, `clock_divider.v:314`) |
| Mutual sync | **none** | `stretch` input hardwired `1'b0` (`iigs.sv:2643`, `// TODO: Connect to VGC stretch signal`) |

Both produce a 912-tick line *on average*, but they start at an arbitrary relative phase and
**nothing pulls them back together**, so any per-cycle jitter (a slow cycle, a refresh cycle,
a mode change) shifts the CPU permanently relative to the beam. A double-buffered program never
notices; a single-buffer beam-racer like textfunk smears progressively down the screen.

### Why this is the bug, per the two reference models

**Real hardware (krue FPI paper, `doc/fpi_timing.md` §PH0 SYNCHRONIZATION):**
> "PH0 is generated by the MEGA II and is **not connected to the FPI**. The FPI **recreates PH0
> internally using 14M and STRETCH, a signal generated by the VGC**." …and during a SYNC cycle
> the FPI "extends PH2 so that it is synchronized to PH0."

There is exactly **one** synchronizing signal — `STRETCH`, emitted by the VGC once per line —
and it drives *both* the Mega II's real PH0/video counters *and* the FPI's internal PH0 copy.
That shared signal is what keeps the CPU's SYNC cycles phase-locked to the beam.

**GSSquared (`software_emulators/gssquared/src/NClock.hpp`):** a *single* shared `c_14M`
counter. The CPU advances it (1–5 ticks/cycle by mode); video position is computed directly
from it. Two couplings keep them locked:
- Per scanline (every 65 video cycles): `extra_per_scanline (=2)` is added to **both**
  `c_14M` and `video_c14m` (lines ~361-365) — the NTSC stretch is shared, not duplicated.
- Every **SYNC** access (banks `$E0/$E1`, `$C0xx` I/O, shadowed writes): the cycle length is
  `14 + (14 - video_cycle_14M_count)` (lines ~310-318) — i.e. the slow access **waits for the
  next PH0 boundary defined by the video counter**, re-aligning the CPU to the beam.

Both models share one truth: **the CPU's PH0/SYNC timing is derived from the same source as
the video position.** Ours derives it from a private counter. That is the defect.

textfunk leans on exactly this: it reads `$C02E` (vsync spin) and `$C02F` (sub-char fine align)
once per frame, then draws the tunnel as a stream of **text writes into `$E0`** — every one of
which is a SYNC cycle. On real hardware (and in GSS) each of those writes re-locks the CPU to
the beam, so the picture stays coherent. In our RTL the SYNC alignment locks to the *private*
`ph0_counter`, not the beam, so the re-lock pulls toward the wrong phase.

---

## 2. Design principle for the fix

Keep the FPGA architecture (video **must** free-run off the pixel clock to drive the display),
but **make the VGC/Mega II horizontal position the single source of PH0 phase**, and tie the
SYNC alignment to it — exactly as the FPI "recreates PH0 from VGC STRETCH." Concretely:

> One Mega II character = 14 pixels = one PH0 period. So `video_timing.v`'s `hsub` (0..13) **is**
> the PH0 phase, and a strobe at the char boundary (`hsub==13 -> 0`) **is** the PH0 edge —
> already including the 2-tick NTSC stretch, because the stretch is absorbed at the line
> boundary in the video counter. Feed that phase/strobe into `clock_divider.v` and delete the
> private `scanline_ph0_ctr`/`fast_stretch` regeneration.

This removes the second timebase entirely: there is no longer a CPU-side line-length model to
drift, because the CPU takes its PH0 boundaries from the beam.

### Why the earlier "PH0-slaved-to-VGC" experiment only half-worked

The reverted experiment exported `hsub` and did `ph0_counter_next = hphase` — a hard jam of the
counter. That fixed **char-level** alignment (`$C02F` went stable and matched GSS offset 6) but
left sub-char drift, and was (wrongly) blamed on the CPU. We now know the CPU is exact, so the
residual was in the *mechanics of the jam*: forcing `ph0_counter` from `hsub` makes individual
PH0 periods irregular when the free-running counter and `hsub` disagree, which corrupts the
fast-cycle/refresh tick accounting that is keyed off `ph0_counter`. The fix is **not** to jam
the counter but to (a) take the PH0 **edge strobe** from video and (b) gate the **SYNC
alignment** on it, letting fast cycles count their 5 ticks but re-phasing at each real PH0 edge.

## 3. Staged plan

Each stage is independently testable and independently revertable. Do **not** advance a stage
until regression + `customtests/selftest0b.2mg` (TEST 0B) + `selftest05.2mg` (speed test 5)
all still pass.

### Stage 0 — Measurement harness (DONE — baseline captured)
Built an objective drift metric so we stop eyeballing screenshots. **Implemented:**
- `Vemu --beam-trace <start>[,<end>]` (`vsim/sim_main.cpp`) logs, per committed CPU cycle, the
  beam position (`V`, `H_CHAR`) + PC/opcode/addr/data to `beam_trace.csv`. (`V`/`H_CHAR` got
  `/*verilator public_flat*/` in `rtl/iigs.sv`.)
- `vsim/beam_drift.py` — frame-to-frame metric: anchors each frame on textfunk's own `$C02F`
  fine-align read, then measures beam spread at PC-aligned instruction indices.
- `vsim/beam_stage0.sh` — one-shot repro: runs vsim + the GSSquared golden and prints both.

**Baseline (textfunk, frames 430-436):**
- vsim render is **97.3% frame-to-frame stable** (≤1 char) with a **1-char `$C02F` jitter**
  (`$5A`↔`$5B`). i.e. the tearing is a **deterministic, stable mis-timing — not flicker.**
  *Consequence:* frame-to-frame self-comparison proves determinism but CANNOT measure absolute
  correctness; the absolute error needs the GSS golden.
- **GSS golden (renders textfunk cleanly):** at the identical instruction `LDA $2F` @ `00/207B`,
  GSS reads `$C02F == $48` (char 9), **rock-stable** (228/230). vsim reads `$5A`/`$5B`
  (char 27/28). **Absolute CPU↔beam phase error ≈ 18 chars (~250 px)** — the deterministic
  mis-timing that garbles the tunnel. (Some constant part may be an H-origin convention diff;
  the 1-char jitter + the garble confirm a real phase error on top.)

**Acceptance for Stages 1-3:** vsim's `$C02F` read at `00/207B` → `$48` (char 9), stable
frame-to-frame (jitter → 0). Re-measure with `./beam_stage0.sh` after each stage.

### Stage 1 — Export a clean PH0 phase/strobe from the video timebase
In `rtl/video_timing.v`, expose:
- `output [3:0] ph0_phase` = `hsub` (0..13), the position within the current Mega II char.
- `output ph0_edge` = 1-tick strobe at the char boundary (the PH0 falling edge).
These are pure combinational/registered taps on existing counters — no behavior change yet.
Wire them up through `rtl/iigs.sv` to `clock_divider` (replace the dead `.stretch(1'b0)`).
**Validate:** build + full regression must be byte-identical (nothing consumes the new ports
yet). This isolates the wiring from the behavioral change.

### Stage 2 — Drive SYNC alignment from the video PH0 edge
In `rtl/clock_divider.v`, in the `slowMem`/`slow` paths, replace the alignment condition that
currently uses the private `ph0_counter_next == 0` with the **video `ph0_edge`** (and the
overlap/`M2SEL` window from `ph0_phase`). This is the core change — it makes every slow access
end on the **beam's** PH0 boundary, matching GSS's `+(14 - video_cycle_14M_count)` and the FPI's
M2SEL overlap.
**Validate:** Stage-0 drift curve should collapse toward flat; TEST 0B + speed test 5 + full
regression must pass; benchmark (`cpu-speed-timing` memory) must stay within ~1%.

### Stage 3 — Retire the private CPU-side stretch
Once SYNC aligns to the beam, the private `scanline_ph0_ctr` + `fast_stretch` regeneration in
`clock_divider.v` (lines ~309-321, 388-403) is redundant and is itself a drift source. Remove
it (or fold it so the only stretch is the one the video already absorbed). The CPU line length
now *is* the video line length by construction.
**Validate:** same gates as Stage 2; confirm 912-tick line is preserved by the video tap, not a
CPU counter.

### Stage 4 — FPGA clock-domain hardening
In sim (`FASTSIM`), `clk_14M == clk_vid == clk_sys` and `ce_pix==1`, so the new taps are
same-domain and free. On the board, `clk_vid` (28M) and `clk_sys` (14M) are phase-locked 2×
from one PLL (see `fpga-test-rig` notes / `Apple-IIgs.sv`), so a clean single-stage sample of
`ph0_phase`/`ph0_edge` into the `clk_14M` domain is sound — but make the CDC explicit and
documented, and re-run on hardware via the test rig.
**Validate:** on-board boot + a beam-racer; compare against sim.

### Stage 5 — Iterate with more demos
With drift bounded and a measurement harness in place, take the next demos the user provides,
add each to the Stage-0 comparison, and tighten any remaining per-mode (SHR vs text) alignment
constants. Expect small residual constant offsets (the GSS "offset 6" fine-align) — calibrate
those against `$C02F` rather than guessing.

---

## 4. Risks & guards
- **Regression gate every stage:** `vsim/regression.sh`, TEST 0B (`selftest0b.2mg`), speed
  test 5 (`selftest05.2mg`), and the BENCHMARKv5 sanity (see `cpu-speed-timing` memory). Stop
  and report on any diff, per CLAUDE.md.
- **Don't chase the CPU.** It is proven exact; any fix that edits `rtl/65C816/` to "fix timing"
  is almost certainly wrong. Confine changes to `clock_divider.v` / `video_timing.v` / the
  `iigs.sv` wiring.
- **gssquared is a reference, not ground truth.** It matches hardware well on the SYNC-align
  model but is not bit-exact; use it to bound the drift curve, then trust real-hardware
  self-tests and the FPI paper for the absolute numbers.
- **Keep it FPGA-friendly:** clock *enables* only, no new free-running clocks, explicit CDC for
  the 28M→14M tap.

---

## 5. One-line summary
Replace the CPU's private, free-running PH0/stretch timebase with the VGC/Mega II horizontal
position (one Mega II char = one PH0 period), and align every SYNC cycle to that beam-derived
PH0 edge — i.e. implement the FPI's "recreate PH0 from VGC STRETCH" / GSSquared's
"wait to the video PH0 boundary" coupling, which our design currently omits.
